# Votre première spec

Un exemple pas à pas. Nous écrivons ensemble un `spec.md` pour une feature **réaliste** — un « toggle dark mode avec sync préférence système » — de la page blanche à l'artefact revuable.

<Aside type="tip">
  La sortie complète vit à [`specs/mvp-launch/spec.md`](https://github.com/aissablk1/speckitlab/tree/main/specs/mvp-launch/spec.md) — c'est la spec **de ce site**.
</Aside>

## Les 8 sections (dans l'ordre)

### 1. Problème

Un paragraphe, concret, avec la *conséquence* de ne pas résoudre.

> Les utilisateurs qui préfèrent les interfaces sombres voient actuellement un flash blanc aveuglant à chaque chargement. Au-delà de l'esthétique, cela provoque une fatigue oculaire lors des sessions du soir et casse la qualité perçue du produit.

### 2. Audience

Un petit tableau. Pas plus de 3-4 personas.

| Persona | Besoin | Fréquence |
|---|---|---|
| Coder du soir | Dark par défaut | Chaque session |
| Utilisateur appareil partagé | Toggle sans login | Occasionnel |

### 3. User stories

Format classique. **Chaque story doit être testable.**

> *En tant que visiteur*, je veux que le site respecte ma préférence OS dark au premier chargement, *afin de* ne pas voir un flash du mauvais thème.

### 4. Critères d'acceptation (AC)

Un tableau. Chaque AC a une colonne **testable-par**. C'est **critique** — Claude Code s'en sert pour s'auto-vérifier.

| # | Critère | Testable par |
|---|---|---|
| AC-1 | Le premier paint respecte la préférence système | Test visuel Playwright |
| AC-2 | Le toggle persiste entre sessions | Manuel + check localStorage |

### 5. Contraintes

Contraintes de stack, budgets perf, minimums d'accessibilité, posture de sécurité. Pas de cérémonial — uniquement ce qui contraint.

### 6. Non-objectifs

**La section la plus sous-utilisée.** Lister explicitement ce qui est *hors* scope. Empêche le scope creep et l'over-engineering.

### 7. Métriques de succès

Quantitatives. Sans ça, impossible de savoir quand c'est terminé.

### 8. Risques

Ce qui peut mal tourner, probabilité, impact, mitigation. 3-5 lignes max.

## Pièges courants



### Écrire des détails d'implémentation dans la spec

« On utilisera `useState` + une variable CSS » → va dans `plan.md`, pas `spec.md`.

### Sauter les non-objectifs

Sans non-objectifs, votre agent va sur-construire. Toujours les lister.

### AC flous

« Devrait bien marcher » est intestable. « LCP < 1,5 s en throttling 3G » est testable.

### Écrire une fois, ne jamais revisiter

Les specs sont des documents vivants. Mettez-les à jour quand la réalité diverge. Datez chaque changement.
