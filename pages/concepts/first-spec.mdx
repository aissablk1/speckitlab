# Your first spec

A walked-through example. We'll write a `spec.md` for a **realistic** feature — a "dark mode toggle with system preference sync" — from blank page to reviewable artefact.

<Callout type="info">
  A complete example of the 8 sections in action — for this very website — is assembled in the [templates](/templates/spec) section.
</Callout>

## The 8 sections (in order)

### 1. Problem

One paragraph, concrete, with the *consequence* of not solving it.

> Users who prefer dark interfaces currently see a blinding white flash on every page load. Beyond aesthetics, this causes eye strain during evening work sessions and breaks the perceived quality of the product.

### 2. Audience

A small table. No more than 3-4 personas.

| Persona | Need | Frequency |
|---|---|---|
| Evening coder | Dark by default | Every session |
| Shared-device user | Toggle without login | Occasional |

### 3. User stories

Classic format. **Each story must be testable.**

> *As a visitor*, I want the site to match my OS dark mode preference on first load, *so that* I don't see a flash of the wrong theme.

### 4. Acceptance criteria (AC)

A table. Each AC has a **testable-by** column. This is **critical** — Claude Code uses these to self-verify.

| # | Criterion | Testable by |
|---|---|---|
| AC-1 | First paint matches system preference | Playwright visual test |
| AC-2 | Toggle persists across sessions | Manual + localStorage check |

### 5. Constraints

Stack constraints, performance budgets, accessibility minimums, security posture. Skip ceremony — list only what constrains.

### 6. Non-goals

**The most under-used section.** Explicitly list what is *not* in scope. Prevents scope creep and unnecessary implementation.

### 7. Success metrics

Quantitative. Without this, you can't know when you're done.

### 8. Risks

What could go wrong, probability, impact, mitigation. 3-5 rows max.

## Common pitfalls

<Steps>

### Writing implementation details in the spec

"We'll use `useState` + a CSS variable" → belongs in `plan.md`, not `spec.md`.

### Skipping non-goals

Without non-goals, your agent will over-engineer. Always list them.

### Fuzzy acceptance criteria

"Should work well" is untestable. "LCP < 1.5s on 3G throttling" is testable.

### Writing once, never revisiting

Specs are living documents. Update them when reality diverges. Date every change.

</Steps>
